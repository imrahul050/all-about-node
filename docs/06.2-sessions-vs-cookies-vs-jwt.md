# Sessions vs Cookies vs JWT

# Introduction

One of the most common backend interview questions is:

```text id="m3l1vf"
What is the difference between:

Cookies
Sessions
JWT
```

Most developers use JWT but cannot explain why it exists or what problem it solves.

Understanding these concepts helps you design authentication systems correctly.

---

# What is a Cookie?

A Cookie is a small piece of data stored in the user's browser.

The server sends the cookie to the browser.

The browser automatically sends it back with future requests.

---

# Cookie Flow

```text id="jlwmql"
Server
   ↓
Set Cookie
   ↓
Browser Stores Cookie
   ↓
Future Requests
   ↓
Cookie Sent Automatically
```

---

# Example Cookie

Server Response:

```http id="vyd2km"
Set-Cookie:
userId=1
```

Browser stores:

```text id="r95xg4"
userId=1
```

and automatically sends it later.

---

# What Can Cookies Store?

Examples:

* Session IDs
* Theme Preferences
* Language Settings
* Authentication Tokens

Cookies are simply storage containers.

---

# Are Cookies Secure?

Not automatically.

Cookies can be:

```text id="brp6vy"
Modified

Stolen

Intercepted
```

unless security settings are used.

---

# Cookie Security Options

Common settings:

```text id="x36cqh"
HttpOnly

Secure

SameSite
```

These improve security significantly.

---

# What is a Session?

A Session is server-side user storage.

Unlike cookies:

```text id="g4ldvc"
Session Data
```

is stored on the server.

The browser only stores a Session ID.

---

# Session Flow

```text id="lpg93y"
Login
   ↓
Create Session
   ↓
Store Session In Server
   ↓
Send Session ID To Browser
   ↓
Browser Sends Session ID
   ↓
Server Finds Session
```

---

# Example

Server Memory:

```text id="c5qdr0"
Session 123

User ID = 1

Role = ADMIN
```

Browser:

```text id="tzyvfm"
sessionId=123
```

The actual data remains on the server.

---

# Session Authentication

During login:

```text id="vgsj4n"
Email
Password
```

verified.

Server creates:

```text id="3gkifk"
Session
```

and stores user information.

Future requests use the session ID.

---

# Advantages of Sessions

Benefits:

* Easy to implement
* User data stays on server
* Easy logout
* More control

---

# Disadvantages of Sessions

Problems:

* Requires server storage
* Memory usage increases
* Harder to scale
* Requires session sharing between servers

---

# What is JWT?

JWT stands for:

```text id="4i2bgj"
JSON Web Token
```

JWT is a stateless authentication mechanism.

Instead of storing session data on the server:

```text id="v1j62z"
User information
```

is stored inside the token.

---

# JWT Flow

```text id="yjlwmc"
Login
   ↓
Generate Token
   ↓
Send Token
   ↓
Store Token
   ↓
Future Requests
   ↓
Verify Token
```

No session storage required.

---

# Session vs JWT

Session:

```text id="jxgsjv"
Server stores user data
```

JWT:

```text id="74fdly"
Token stores user data
```

This is the biggest difference.

---

# Example Session Authentication

Server:

```text id="7w4v0u"
Session #123
User ID = 1
```

Browser:

```text id="2kv7k8"
sessionId=123
```

Every request:

```text id="lz0dpa"

```

required.

---

# Example JWT Authentication

Token:

```text id="fq4a1m"
{
  id: 1,
  role: "AUTHOR"
}
```

stored inside JWT.

Server verifies token.

No database lookup for session required.

---

## Stateless

# Most Important Interview Questions

Q: What is a Cookie?

Answer:

A Cookie is a small piece of data stored in the browser and automatically sent with future requests to the same server.

---

Q: What is a Session?

Answer:

A Session is server-side storage that maintains user state. The browser stores only a session identifier.

---

Q: What is JWT?

Answer:

JWT is a stateless authentication mechanism that stores user information inside a signed token rather than server-side session storage.

---

Q: Difference Between Sessions and JWT?

Answer:

Sessions store user state on the server.

JWT stores user information inside the token itself.

Sessions are stateful.

JWT is stateless.

---

Q: What is Stateless Authentication?

Answer:

Stateless authentication means the server does not store user session information. Each request contains all information needed for authentication.

---

Q: Why JWT for APIs?

Answer:

JWT is scalable, stateless, and works well across multiple servers, mobile applications, and microservices.

---

Q: Is JWT Always Better Than Sessions?

Answer:

No.

Sessions are often simpler for traditional applications, while JWT is better suited for distributed and API-driven architectures.

---

# Senior Interview Answer

Cookies, Sessions, and JWT serve different purposes. Cookies are a browser storage mechanism. Sessions provide server-side authentication state management. JWT provides stateless authentication by embedding user information inside a signed token. The choice depends on scalability requirements, architecture, and security considerations rather than one technology being universally superior.
